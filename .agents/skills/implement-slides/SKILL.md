# Skill: `implement-slides`

This skill takes a slide specification plan generated by `planning-slides` and builds the presentation HTML file using the **Reveal.js + dist/main.css + dist/main.js** component system of Universidad Icesi, automatically exporting the final **PDF slide deck** (`slides/<topic>/<topic>.pdf`).

---

## Parameters & Usage

- **`spec_file_path`**: Absolute or relative path to the plan specification markdown file saved under `.agents/spec/slides-plan/`.

Example usage:
```bash
/implement-slides .agents/spec/slides-plan/nestjs-plan.md
```

---

## Execution Guidelines & Workflow

```
1. Validate Spec & Pre-Critique  ──►  2. Create Custom SVGs & Icons  ──►  3. Write HTML Presentation
                                                                               │
                                      4. Export PDF Deck (Decktape)  ◄─────────┘
```

---

### 1. Read & Validate Specification with Pre-Implementation Critique

Before writing any HTML, the agent MUST perform an **agentic critique pass**:

#### Pre-Implementation Checklist:
| Check | Action if FAIL |
|---|---|
| **No external redirects** | Spec contains "Taller Práctico", "Lab 01", "Workshop", "Hands-on Exercise" → **Replace** with self-contained conceptual or deep-dive slides on the same topic. |
| **Self-contained content** | Slide lacks educational payload → **Expand** with detailed explanations, code snippets, or architectural diagrams. |
| **Function signature match** | Check parameter counts against catalog (Section 3A) → **Correct** signature call. |
| **No emojis in icon slots** | Spec contains emoji (🚀, ⚙️, 📦, etc.) → **Replace** with inline SVG or custom vector asset (`slides-iconification`). |
| **Graphic strategy choice** | **Mermaid diagrams fail in PDF exports via Decktape (they render as white boxes).** → **Replace** all Mermaid diagram specs with **custom inline SVG graphics (`slides-svg-graphics`)**. |

---

### 2. File & Directory Setup
1. Destination directory: `slides/<topic>/`.
2. Assets directory: `slides/<topic>/assets/`.
3. Create all custom SVGs, icons (`slides-iconification`), and vector graphics (`slides-svg-graphics`) before HTML assembly.

### 3. HTML Slide Generation Rules

#### A. Component Catalog Signatures (Mandatory)
Use ONLY functions from `dist/main.js`. Verify argument count matches exactly:

| Function | Args | Usage Notes |
|---|---|---|
| `icesi.titleSlideA(title, subtitle)` | 2 | Main cover slide. Subtitle yellow box accepts `<span class="slide-footer-tag">` for faculty labels. |
| `icesi.titleSlideF(title, subtitle)` | 2 | Visual break for major module shifts (never for external workshop redirects). |
| `icesi.slideStandard(title, content)` | 2 | Single content area slide. |
| `icesi.slideTwoCols(title, col1, col2)` | 3 | Two column split layout. |
| `icesi.slideThreeCols(title, col1, col2, col3)` | 4 | Three column split layout. |
| `icesi.slideFourCards(title, c1t, c1b, c2t, c2b, c3t, c3b, c4t, c4b)` | 9 | 4-card grid. Card headers (`c1t`) are small 14px uppercase labels. |
| `icesi.slideSidebarLeftBlue(title, content, sidebarVisual)` | 3 | Left blue sidebar. `sidebarVisual` accepts SVG graphics, icons, or Mermaid. |
| `icesi.slideSidebarLeftOrange(title, content, sidebarVisual)` | 3 | Left orange sidebar. |
| `icesi.slideSidebarLeftPurple(title, content, sidebarVisual)` | 3 | Left purple sidebar. |
| `icesi.slideStripeTopLeft(title, content)` | 2 | Top left accent banner slide. |
| `icesi.slideStripeTopRight(title, content)` | 2 | Top right accent banner slide. |
| `icesi.sectionSlideA(title)` | **1** | Standard section divider (bottom blue block). |
| `icesi.sectionSlideC(title)` | **1** | Major chapter divider. |
| `icesi.sectionSlideE(title, content)` | **2** | Minimal white divider. Title supports `<br><small style="font-size:0.5em;">`. Optional content for diagrams. |
| `icesi.sectionSlideEBlue(title, content)` | **2** | Blue section divider. Optional content for diagrams. |
| `icesi.sectionSlideEGreen(title, content)` | **2** | Green section divider. Optional content for diagrams. |
| `icesi.sectionSlideEYellow(title, content)` | **2** | Yellow section divider. Optional content for diagrams. |
| `icesi.sectionSlideEOrange(title, content)` | **2** | Orange section divider. Optional content for diagrams. |
| `icesi.mermaid(code)` | 1 | Inline Mermaid diagram container. |

#### B. Graphics Strategy: SVG vs. Mermaid
- **CRITICAL - PDF EXPORT LIMITATION**: **Mermaid diagrams fail to render in PDF exports (they appear as empty white boxes)** because Decktape captures the screen before the async JS finishes.
- **MANDATORY**: You MUST convert any requested Mermaid diagrams into **Custom SVG Vector Graphics (`slides-svg-graphics`)** using inline SVGs (`{ type: 'graphic', html: '<svg>...</svg>' }`).
- **SVG Dimension Rule**: ANY inline `<svg>` code generated by you MUST include explicit `width` and `height` attributes (e.g. `width="400px" height="400px"`) to prevent them from breaking the CSS layout and overlapping text.

#### C. Absolute Prohibition of Emojis
- **NEVER use emoji characters** (🚀, ⚙️, 📦, ⏱️, 🔌, etc.) anywhere in slides.
- Use compact semantic inline SVGs (`slides-iconification`) inside `.icon-symbol`, card headers, and sidebar lists.

#### D. Self-Contained Pedagogical Requirement
- Every slide must provide standalone educational value with explanations, code examples, or visual diagrams.
- No slides directing students to external activities or workshops.

#### E. Spacing and Text Sizing Rules
- **Paragraphs below titles or lists**: Keep font-size between `16px` and `19px`. Use `margin-top: 12px; line-height: 24px;` so text never overflows the slide content box (max height ~460px).
- If content has > 5 bullet points or long code blocks, split into 2 slides or use a two-column layout.

#### F. Implementing "Frames" (Progressive Reveal)
- If the spec plan specifies "Frames" or progressive reveal (e.g., showing 1 item, then 2, then 3), you MUST implement this by **creating completely separate, consecutive slides** in the `slides` array.
- Do NOT use Reveal.js fragments (`class="fragment"`). They do not always export reliably to separate pages in Decktape PDF generation. Just duplicate the slide function call and add the next item.

### 4. PDF Generation & Final Deliverable

> **Note: Do NOT generate PNG screenshots or screenshot image files.**

#### Step 4.1: (Optional) Ensure Local Server is Running
A local server is not strictly needed for PDF generation but is useful for viewing: `npx -y http-server . -p 8080`.

#### Step 4.2: Export PDF File via Decktape
Export the presentation HTML to the PDF deliverable (`slides/<topic>/<topic>.pdf`) using local file paths. Since Decktape can read from `file:///`, we do not need the HTTP server for this step:
```powershell
npx -y decktape reveal "file://$PWD/slides/<topic>/<topic>.html" "slides/<topic>/<topic>.pdf" --size 1280x720
```
*(Note: Replace $PWD with the absolute path to the directory if using Windows CMD/Powershell, e.g. `file://C:/ruta/al/proyecto/slides...`)*

#### Step 4.3: Final Quality Inspection
Verify the generated HTML and PDF files:
- [ ] `slides/<topic>/<topic>.html` exists and loads cleanly without JS errors.
- [ ] `slides/<topic>/<topic>.pdf` is generated successfully.
- [ ] All function signatures match `dist/main.js`.
- [ ] Zero emojis are present.
- [ ] All slides are self-contained without external workshop redirects.
- [ ] **CODE BLOCKS INDENTATION**: Ensure that any custom CSS for `.code-box` or code snippets explicitly includes `white-space: pre-wrap;` or `white-space: pre;` so that formatting and line breaks in template literals do not collapse into a single line. Alternatively, use `<pre><code>` tags.
