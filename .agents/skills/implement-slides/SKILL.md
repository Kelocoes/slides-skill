# Skill: `implement-slides`

This skill takes a slide specification plan generated by `planning-slides` and builds the presentation HTML file using the **Reveal.js + dist/main.css + dist/main.js** component system of Universidad Icesi, automatically exporting the final **PDF slide deck** (`slides/<topic>/<topic>.pdf`) and conducting a mandatory **multimodal visual screenshot inspection pass** (`slides/<topic>/build/`).

---

## 📥 Parameters & Usage

- **`spec_file_path`**: Absolute or relative path to the plan specification markdown file saved under `.agents/spec/slides-plan/`.

Example usage:
```bash
/implement-slides .agents/spec/slides-plan/nestjs-plan.md
```

### 1. Read & Validate Specification with Pre-Implementation Critique

Before writing any HTML, the agent MUST perform an **agentic critique pass**:

#### Pre-Implementation Checklist:
| Check | Action if FAIL |
|---|---|
| **No external redirects** | Spec contains "Taller Práctico", "Lab 01", "Workshop", "Hands-on Exercise" → **Replace** with self-contained conceptual or deep-dive slides on the same topic. |
| **Self-contained content** | Slide lacks educational payload → **Expand** with detailed explanations, code snippets, or architectural diagrams. |
| **Function signature match** | Check parameter counts against catalog (Section 4A) → **Correct** signature call. |
| **No emojis in icon slots** | Spec contains emoji (🚀, ⚙️, 📦, etc.) → **Replace** with inline SVG or custom vector asset. |
| **Graphic strategy choice** | For complex architecture/flows → prefer **custom inline SVG graphics (`slides-svg-graphics`)** over plain text or brittle Mermaid strings for 100% layout control. |

### 3. HTML Slide Generation Rules

#### A. Component Catalog Signatures (Mandatory)
Use ONLY functions from `dist/main.js`. Verify argument count matches exactly:

| Function | Args | Usage Notes |
|---|---|---|
| `icesi.titleSlideA(title, subtitle)` | 2 | Main cover slide. Subtitle yellow box accepts `<span class="slide-footer-tag">` for faculty labels. |
| `icesi.titleSlideF(title, subtitle)` | 2 | Visual break for major module shifts (never for external workshop redirects). |
| `icesi.slideStandard(title, content)` | 2 | Single content area slide. |
| `icesi.slideTwoCols(title, col1, col2)` | 3 | Two column split layout. |
| `icesi.slideThreeCols(title, col1, col2, col3)` | 4 | Three column split layout. |
| `icesi.slideFourCards(title, c1t, c1b, c2t, c2b, c3t, c3b, c4t, c4b)` | 9 | 4-card grid. Card headers (`c1t`) are small 14px uppercase labels. |
| `icesi.slideSidebarLeftBlue(title, content, sidebarVisual)` | 3 | Left blue sidebar. `sidebarVisual` accepts SVG graphics, icons, or Mermaid. |
| `icesi.slideSidebarLeftOrange(title, content, sidebarVisual)` | 3 | Left orange sidebar. |
| `icesi.slideSidebarLeftPurple(title, content, sidebarVisual)` | 3 | Left purple sidebar. |
| `icesi.slideStripeTopLeft(title, content)` | 2 | Top left accent banner slide. |
| `icesi.slideStripeTopRight(title, content)` | 2 | Top right accent banner slide. |
| `icesi.sectionSlideA(title)` | **1** | Standard section divider (bottom blue block). |
| `icesi.sectionSlideC(title)` | **1** | Major chapter divider. |
| `icesi.sectionSlideE(title)` | **1** | Minimal white divider. Title supports `<br><small style="font-size:0.5em;">`. |
| `icesi.sectionSlideEBlue(title)` | **1** | Blue section divider. |
| `icesi.sectionSlideEGreen(title)` | **1** | Green section divider. |
| `icesi.sectionSlideEYellow(title)` | **1** | Yellow section divider. |
| `icesi.sectionSlideEOrange(title)` | **1** | Orange section divider. |
| `icesi.mermaid(code)` | 1 | Inline Mermaid diagram container. |

#### B. Graphics Strategy: SVG vs. Mermaid
- **Prefer Custom SVG Vector Graphics (`slides-svg-graphics`)**: For architecture diagrams, tech stacks, layered models, and sidebar visuals, use `{ type: 'graphic', html: '<svg>...</svg>' }` or inline SVGs. Custom SVGs ensure **100% pixel-perfect rendering**, custom typography, and zero layout breakage.
- **When using Mermaid (`slides-mermaid-diagrams`)**: Ensure `window.mermaid.run()` is invoked in `icesi.init()` and pass `{ type: 'mermaid', code: '...' }` to sidebars.
- **Never render raw unparsed Mermaid code text** on slides.

#### C. Absolute Prohibition of Emojis
- **NEVER use emoji characters** (🚀, ⚙️, 📦, ⏱️, 🔌, etc.) anywhere in slides.
- Use compact semantic inline SVGs (`slides-iconification`) inside `.icon-symbol`, card headers, and sidebar lists.

#### D. Self-Contained Pedagogical Requirement
- Every slide must provide standalone educational value with explanations, code examples, or visual diagrams.
- No slides directing students to external activities or workshops.

#### E. Spacing and Text Sizing Rules
- **Paragraphs below titles or lists**: Keep font-size between `16px` and `19px`. Use `margin-top: 12px; line-height: 24px;` so text never overflows the slide content box (max height ~460px).
- If content has > 5 bullet points or long code blocks, split into 2 slides or use a two-column layout.

### 4. PDF Generation & Mandatory Visual QA Screenshot Pass

> **⚠️ MANDATORY STEP: You MUST execute this pass for every presentation.**

#### Step 4.1: Start HTTP Server & Export PDF
1. Start local server: `npx -y http-server . -p 8080` (if not already running).
2. Export PDF via decktape:
   ```powershell
   npx -y decktape reveal http://localhost:8080/slides/<topic>/<topic>.html slides/<topic>/<topic>.pdf --size 1280x720
   ```

#### Step 4.2: Capture Visual QA Screenshots (Playwright)
Use Playwright to navigate through all slides and save PNG screenshots in `slides/<topic>/build/`:

```javascript
// Example Playwright iteration script
const fs = require('fs');
if (!fs.existsSync('slides/<topic>/build')) fs.mkdirSync('slides/<topic>/build', { recursive: true });

// Navigate to presentation, iterate through all slides, capture slide_01.png, slide_02.png, ...
```

#### Step 4.3: Visual Quality Inspection Checklist
Inspect every screenshot saved in `slides/<topic>/build/` against this checklist:

| Check | Pass Criteria |
|---|---|
| **Text Overflow** | Zero text overflowing past slide boundaries or container boxes. |
| **Diagram Visibility** | All diagrams (SVG / Mermaid) are fully rendered and clearly legible. No raw Mermaid code text visible! |
| **Sidebar Content** | Sidebar panel contains visible graphics, SVGs, or icon lists (not empty or broken). |
| **No Emojis** | Zero emoji characters visible on any slide. |
| **No Workshop Redirects** | Zero slides directing to external workshops/labs. |
| **Typography Hierarchy** | Slide Title (38px) > Sub-heading (h3 22px) > Body (19px) > Card Label (14px). |
| **Spacing & Padding** | Adequate vertical spacing between title, paragraphs, and cards. No overlapping elements. |
| **Cover Badge** | Institution label is rendered as a blue badge with white text (`.slide-footer-tag`). |

#### Step 4.4: Refine & Re-verify
If ANY slide fails the inspection checklist:
1. Fix the HTML / CSS / SVG in `slides/<topic>/<topic>.html`.
2. Re-bundle JS/CSS if needed (`npm run build`).
3. Re-export PDF (`slides/<topic>/<topic>.pdf`).
4. Re-capture screenshots and re-inspect until 100% of slides pass.
